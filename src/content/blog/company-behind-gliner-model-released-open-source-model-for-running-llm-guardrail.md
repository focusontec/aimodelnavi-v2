LLM（大規模言語モデル）を実務に導入する際、最大の懸念となるのが「安全性」です。有害な出力を防ぎ、悪意あるプロンプトを遮断する「ガードレール」の構築は不可欠ですが、これまで主流だったガードレールモデルは、数億から数百億のパラメータを持つデコーダー型LLMに基づいたものが多く、推論コストとレイテンシが大きな課題となっていました。

そんな中、GLiNERモデルの開発チーム（Pioneer AI / Fastino Labs）から、オープンソースの小型安全モデレーションモデル**「GLiGuard」**がリリースされました。

## 「生成」から「分類」へ：アーキテクチャの転換による劇的な高速化

従来のLlamaGuardやShieldGemmaなどのガードレールモデルは、デコーダーのみのTransformerアーキテクチャを採用しています。これらは「安全かどうか」をテキスト生成（オートリグレッシブ）によって判定するため、1トークンずつ順番に処理する必要があり、特に複数の安全基準を評価する場合にレイテンシが蓄積します。

対してGLiGuardは、**3億（300M）パラメータのエンコーダーベースモデル**です。安全性の判定を「テキスト生成問題」ではなく「テキスト分類問題」として再定義しました。入力テキストとタスク定義（ラベル）を同時にエンコードし、一度のフォワードパスで全てのラベルをスコアリングします。これにより、評価基準を増やしてもレイテンシが増加しないという画期的な特性を持っています。

## 1回のパスで4つのタスクを同時処理

GLiGuardは以下の4つのモデレーションタスクを同時に処理可能です：

1. **安全性分類 (Safe / Unsafe)**：ユーザープロンプトとモデル応答の両方を判定。
2. **ジェイルブレイク戦略の検知 (11種類)**：プロンプトインジェクションやロールプレイなど、回避策の検知。
3. **有害カテゴリの検知 (14種類)**：暴力、ヘイトスピーチ、PII（個人情報）露出など、具体的な有害性の特定。
4. **拒絶検知 (Compliance / Refusal)**：モデルが正しく要求を拒絶したか、あるいは不当に拒絶（過剰拒絶）したかを判定。

## ベンチマーク結果：サイズを凌駕する精度と速度

驚くべきは、その軽量さでありながら、数十倍のサイズを持つモデルに匹敵する精度を叩き出している点です。

- **精度 (Accuracy)**：9つの安全性ベンチマークにおいて、GLiGuardは23倍〜90倍のサイズを持つモデルと同等、あるいはそれを上回る精度を記録しました。
    - **プロンプト分類**：平均F1スコア **87.7**（最高モデルPolyGuard-Qwenの89.4にわずか1.7ポイント差）。
    - **レスポンス分類**：平均F1スコア **82.7**（Qwen3Guard-8Bに次いで2番目の高スコア）。
    - **比較対象**：LlamaGuard4 (12B)、ShieldGemma (27B)、NemoGuard (8B) などの巨大モデルを凌駕的に効率的なサイズで上回っています。
- **速度 (Speed)**：従来のデコーダーベースモデルと比較して**最大16倍の高速化**を実現しています。

## まとめ：本番環境への導入が現実的に

GLiGuardはApache 2.0ライセンスでHugging Face Hubに公開されており、誰でも利用可能です。これまで「精度のために速度を犠牲にする」か「速度のために簡易的なフィルタを使う」かの選択を迫られていたエンジニアにとって、高精度かつ低レイテンシなGLiGuardは、リアルタイムな安全対策を実装するための強力な武器になるでしょう。
